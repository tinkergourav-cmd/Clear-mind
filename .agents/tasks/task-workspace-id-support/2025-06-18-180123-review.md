# Workspace ID ownership stamping across object creation, duplication, and import

This change introduces a `workspaceId` field on every workspace-level object (nodes, edges, groups, pins) to establish explicit ownership. The implementation touches all creation flows (addNode, createGroup, addPin, edge creation via connection ports and keyboard shortcut), all duplication/clone flows (duplicateNode, cloneNode, pasteNode, pasteGroup, bulk duplicate, cloneNodeToWorkspace), a new `duplicateWorkspace` function with full ID remapping, import/export flows (handleImport, importAllData, partial import), a migration layer at initialization and project switch, and updates `duplicateProject` to generate fresh workspace IDs.

Watch for: Images are excluded from workspaceId stamping — they're never migrated, never stamped at creation, and not stamped in the `duplicateWorkspace` clone (confirmed). The migration layer is duplicated verbatim in four places rather than factored into a helper (confirmed). The `duplicateWorkspace` function's group ID generation uses `Date.now()` inside a synchronous forEach, meaning groups duplicated in the same millisecond will share a timestamp prefix (likely, though the `idCounter` suffix prevents actual collision).

## High-level view

The core approach stamps `workspaceId: activeTab` (or the target workspace ID) on every object at the point of creation, and runs a migration pass on loaded data to backfill objects that predate this feature. The field serves as a denormalized ownership marker — redundant with the workspace's own array containment, but useful for future cross-workspace queries or integrity checks.

Images are the gap. Every other object type gets workspaceId treatment across creation, duplication, migration, and import — but images are omitted from all of these. If workspaceId is meant to establish universal ownership, images should be included.

The migration code (the `workspaces.map(ws => ...)` block that stamps missing workspaceIds) appears identically in four places: init, project switch, project delete fallback, and partial import. A bug fix or schema change in one copy won't propagate to the others without manual effort.

## Images: the omitted object type

The implementation explicitly targets nodes, edges, groups, and pins in every path. Images are absent from:

- Migration at init (line 768): stamps nodes, edges, groups, pins — not images
- `duplicateWorkspace` (line 2052): images get new IDs but no `workspaceId`
- Image creation via drag-drop (line 3427): no `workspaceId` field
- `handleImport` (line 2619): stamps nodes, edges, groups, pins — not images
- `importAllData` (line 2674): same pattern, images excluded
- Bulk duplicate in selection menu (line 6336): stamps nodes and edges, images get no `workspaceId`

If images are deliberately excluded because they're treated as workspace-scoped blobs rather than first-class graph objects, the intent should be documented.

## Migration duplication

The migration block:

```javascript
targetWorkspaces = targetWorkspaces.map(ws => ({
  ...ws,
  nodes: (ws.nodes || []).map(n => n.workspaceId ? n : { ...n, workspaceId: ws.id }),
  edges: (ws.edges || []).map(e => e.workspaceId ? e : { ...e, workspaceId: ws.id }),
  groups: (ws.groups || []).map(g => g.workspaceId ? g : { ...g, workspaceId: ws.id }),
  pins: (ws.pins || []).map(p => p.workspaceId ? p : { ...p, workspaceId: ws.id })
}));
```

This appears verbatim at four call sites. Three of them use a conditional check (`n.workspaceId ? n : ...`) while `handleImport` unconditionally overwrites. This inconsistency means `handleImport` does unnecessary object allocation on data that already has correct workspaceIds. More importantly, if a fifth migration point is needed (e.g., a future "merge workspaces" flow), the pattern has to be copy-pasted again. A `stampWorkspaceIds(workspaces)` utility would reduce this to a single call at each site.

## duplicateWorkspace: ID generation strategy

The function uses a hybrid approach: nodes get sequential numeric IDs from `nextId`, while groups, pins, and edges embed `Date.now()`. Since the entire function executes synchronously within a single event loop tick, every `Date.now()` call returns the same millisecond value. The actual uniqueness comes from the `idCounter` suffix.

Group IDs in a duplicated workspace look like `g-1718729000000-42`, `g-1718729000000-43` — the timestamp prefix is noise that suggests temporal ordering but carries no semantic value. Using just the counter (e.g., `g-${idCounter}`) would produce shorter, less misleading IDs without sacrificing uniqueness.
