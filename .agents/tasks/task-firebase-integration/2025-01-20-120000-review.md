# Firestore dual-persistence for mind-mapping projects

The app gains a Firebase Firestore backend as a cloud persistence layer alongside the existing localStorage. A new `firebase.js` module handles initialization with placeholder credentials and a guard (`isFirebaseConfigured`) that keeps the app inert until real keys are supplied. A companion `firebaseService.js` wraps Firestore reads and writes for three data slices (projects array, active project ID, default project ID). `App.jsx` now loads from Firestore first on init, writes to Firestore on every debounced save and project switch, and shows a sync-status icon in the toolbar.

**Watch for:** All user data (including hashed passwords) is written to a single Firestore document with no authentication or security rules mentioned (confirmed). The single-document design will hit the 1 MiB Firestore document size limit as workspaces grow (confirmed). Rapid saves race against each other because the debounce timer resets but in-flight Firestore writes are not cancelled (likely).

## High-level view

The persistence model is write-both, read-primary-fallback-secondary: on load the app attempts Firestore first and falls back to localStorage; on save it always writes localStorage immediately and fires Firestore in the background. The fallback is graceful and the placeholder-detection pattern means no code path runs network calls until the user pastes real credentials.

All project data (nodes, edges, groups, tasks, reminders, passwords) is serialized as a single JSON string inside one Firestore document. This avoids complex collection modeling but introduces a hard ceiling at Firestore's 1 MiB document limit and means every keystroke that triggers a save replaces the entire blob.

There is no Firebase Authentication, no Firestore security rules, and no mention of per-user scoping. Anyone who discovers the project's Firebase config (shipped in the client bundle) can read and overwrite the entire dataset. The project passwords that protect individual workspaces are hashed client-side, but the full project array including those hashes is stored unguarded in Firestore.

The sync-status indicator provides visual feedback (cloud icon, spinner, error state) but no user-actionable recovery: on error the user sees a red icon with no retry mechanism or explanation.

## Single-document data model and Firestore limits

All project state lives in `appData/main` as a `projects` field containing a `JSON.stringify`'d array. Firestore documents cap at 1 MiB. A user with several workspaces, large node content, and embedded images (the app supports image cards) will approach that ceiling. Once hit, `setDoc` will throw and every subsequent save will fail silently (the catch sets `syncStatus` to `'error'` but doesn't surface the cause). Because localStorage has a ~5-10 MB quota and is unaffected, the failure is invisible until the user switches devices and finds stale data.

The serialization choice also means Firestore's query, indexing, and real-time subscription capabilities are unusable — the data is opaque to Firestore. If multi-device sync or conflict resolution is ever needed, this structure makes it nearly impossible without a rewrite.

## No authentication and exposed credentials

`firebase.js` ships credentials directly in the client bundle. The `isFirebaseConfigured` guard only controls whether code runs locally — it does nothing about who can access the Firestore database remotely. Without Firebase Authentication and matching security rules, any client that imports the same config object (or just copies it from the built JS) can call `getDoc`/`setDoc` on `appData/main` and read or replace all user data.

The projects blob includes hashed workspace passwords. While they're SHA-256 hashed client-side, they're still sensitive: an attacker with read access to the document can attempt brute-force recovery or simply overwrite the blob with passwords removed.

## Write-race in fire-and-forget saves

The auto-save effect debounces at 500 ms, then fires `saveProjects(currentProjects)` without awaiting the previous write's completion. If the user triggers two saves 600 ms apart, both network requests fly concurrently. Firestore `setDoc` with `merge: true` is last-write-wins at the document level. If the first write is slow (network jitter), it can land after the second write, reverting data.

Additionally, `saveActiveProject` and `saveDefaultProject` are fired in parallel alongside `saveProjects` during the main save effect. These use `merge: true` on the same document, which means all three writes race against each other at the field level — acceptable for independent fields, but a subtle correctness concern if one write's `merge` partially overlaps with another's in-flight `setDoc`.

## Sync-status indicator without recovery

The toolbar shows four states (synced, syncing, error, offline) via a cloud icon. The error state (`CloudOff` in red) offers no tooltip beyond "Sync error" and no click handler. A user who sees persistent red has no in-app way to retry or diagnose the issue. For a user who explicitly stated "I don't know coding," this gap in UX is material — they won't be able to inspect the console.

## Offline edits lost on reload

During initialization, if Firestore returns data, the code immediately writes it into localStorage. If the user had newer data in localStorage (e.g., they made changes while offline and then regained connectivity), the Firestore data blindly overwrites it. There's no conflict detection or timestamp comparison — Firestore always wins on load, which can silently discard offline edits.
